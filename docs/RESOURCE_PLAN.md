# 资源获取计划

日期：2026-05-21

## 目标

我们先小范围获取资源，跑通 `GushenAgents` 的最小验证闭环。第一阶段只需要支持收盘后研究、观察清单、模拟盘和复盘，不需要实盘下单资源。

## 最小资源清单

### 1. 行情数据

用途：

- 生成每日成交额前 100 股票池。
- 计算趋势、量价、波动、流动性、涨跌停距离。
- 支撑回测和模拟盘。

第一阶段需要：

- A 股日线 OHLCV。
- 成交额。
- 股票基础信息。
- 交易日历。
- 停牌状态。
- ST/退市风险状态。
- 涨跌停状态或涨跌停价格。

建议获取方式：

- 第一阶段以 AKShare 作为主数据入口。
- 优先使用 AKShare 已覆盖的 A 股接口：股票列表、交易日历、日线行情、实时行情、ST 股票、停复牌、涨停池/跌停池。
- 如果某个 AKShare 上游站点短时不可用，先换 AKShare 内部的同类接口或降级到缓存数据，不急着切换数据源。
- Tushare Pro、券商数据或商业数据源只作为第二阶段补充。

验收标准：

- 能稳定生成最近 3 年每个交易日的成交额前 100 股票池。
- 股票代码、交易日、复权、停牌和 ST 状态可追溯。
- 每次数据更新有日志和版本记录。

### 2. 文本数据

用途：

- EventAnalystAgent 识别公告、新闻和风险事件。
- ReviewAgent 解释信号失效原因。

第一阶段需要：

- 交易所公告。
- 上市公司公告标题和摘要。
- 少量高质量新闻源。

建议获取方式：

- 先使用公开公告源和手工小样本。
- 不急着接大规模新闻流。
- 新闻只做风险提示，不做直接买卖依据。

验收标准：

- 能按股票代码和日期检索公告。
- Agent 能输出事件类型、风险等级和证据。
- 输出结构化字段，而不是一段不可落表的文字。

### 3. AI 模型资源

用途：

- 结构化抽取。
- 多空观点生成。
- 风险解释。
- 复盘总结。
- 因子和策略假设生成。

第一阶段需要：

- 一个高推理模型：用于多 Agent 综合判断和复杂复盘。
- 一个低成本模型：用于公告摘要、新闻摘要、字段抽取。
- 支持结构化输出。
- 支持较长上下文。

资源控制：

- 所有 Agent 调用必须记录输入摘要、输出、模型名、时间和成本。
- 同一批股票先只跑前 20 或前 30，等流程稳定再扩到前 100。
- 可缓存公告摘要、特征解释和复盘结论。

验收标准：

- 同样输入能得到格式稳定的 JSON 输出。
- 风控 Agent 能一票否决高风险候选。
- AI 输出能追溯到特征、公告或回测结果。

### 4. 计算和存储

用途：

- 存储行情、特征、Agent 输出和回测结果。
- 本地运行小规模模型实验和回测。

第一阶段建议：

- 本地 Windows 环境即可。
- Python 3.11+。
- SQLite + WAL。先不用 DuckDB；当前阶段更重视小规模频繁读写、快速落表和本地调试。
- 如果后续特征矩阵变大，再把大块行情和特征旁路为 Parquet 文件。
- 20GB 以上可用磁盘。
- 16GB 内存更舒服。

验收标准：

- 全流程可在本机运行。
- 3 年日线数据和前 100 股票池回测能在可接受时间内完成。
- 生成的报告、日志、缓存可清理和复现。

### 5. 资金账户与交易接口

第一阶段不接真实下单接口。

现在只需要：

- 账户资金规模的风险预算假设。
- 交易费用、滑点和最小成交单位配置。
- 模拟盘账户。

后续再考虑：

- QMT、Ptrade 或券商 API。
- vn.py gateway。
- 自动下单前的人工确认和风控拦截。

验收标准：

- 模拟盘连续运行至少 60 个交易日。
- 每笔模拟交易都有入场、出场、仓位、失效条件和复盘。
- 实盘前能证明策略没有依赖未来函数、错误复权或不可成交价格。

## 获取顺序

1. AKShare 接口健康检查。
2. 行情日线和股票基础信息。
3. 交易日历、停牌、ST、涨跌停状态。
4. Agent 结构化输出所需的 AI API。
5. 公告小样本。
6. 回测和模拟盘日志存储。
7. 新闻源和更丰富的文本数据。
8. 资金账户接口。

## 小范围验证范围

第一轮只验证：

- 最近 6 个月数据。
- 每日成交额前 30 股票，而不是前 100。
- 只计算日线特征。
- 只生成观察清单，不做模拟买卖。
- 每天最多输出 5 个候选。

通过后扩大到：

- 最近 3 年数据。
- 每日成交额前 100。
- 模拟盘。
- 周度复盘。
